# Lane4 Issue

이슈 발생 시 관련된 로직을 조회하고 수정 및 해결 방안을 제안한다.

## 이슈 유형별 탐색 전략

### API 에러 (500, 400 등)
1. 에러가 발생한 API 엔드포인트 → 해당 Controller → Service 순으로 추적
2. 요청 파라미터 검증 로직 확인

### 데이터 정합성 이슈
1. lane4-mysql MCP로 실제 데이터 상태 조회
2. 데이터를 생성/수정하는 로직 역추적

### 배치/스케줄러 이슈
1. 스케줄러 설정 및 실행 로직 확인
2. 최근 실행 로그/결과 확인

### 연동 이슈 (외부 API, Firebase, Redis 등)
1. 연동 클라이언트 설정 확인
2. 타임아웃, 인증 정보 확인

## 핵심 제약사항
- `lane4-*` 프로젝트를 돌며 관련 이슈가 발생했을 부분을 탐색한다.
- 정확하지 않은 문제 원인은 전달하지 않는다.
- 수정 방안, 해결 방안을 절대 직접 수정하지 않는다.
- 사용자가 전달한 수정 및 해결 방안에 대해 처리를 요청했을 경우에만 수정한다.

## 탐색 우선순위
  1. 사용자가 언급한 키워드로 관련 프로젝트 특정
  2. API 이슈 → lane4-*-api 우선
  4. 프론트 이슈 → lane4-* 우선

## MCP 활용 기준

  | 상황 | 사용할 MCP | 확인 사항 |
  |------|-----------|----------|
  | 데이터 값이 이상할 때 | lane4-mysql | 실제 DB 데이터 상태 조회 |
  | 캐시 관련 이슈 | redis MCP | 캐시 키 존재 여부, TTL, 값 확인 |
  | 검색/로그 이슈 | elasticsearch MCP | 인덱스 매핑, 검색 쿼리 확인 |
  | 푸시/알림 이슈 | firebase MCP | 토큰 상태, 발송 기록 확인 |

## 이슈 원인 파악 구조
```
원인: <원인 한 줄 정리>
상세: <원인 발생 상세>

해결방안
1. <여러 가지가 있는 경우 순서를 매겨 정리한다.>
```

## 필수 확인 정보
- 에러 메시지 또는 로그 (있는 경우)
- 발생 환경 (개발/스테이징/운영)
- 발생 시점 및 재현 조건
- 영향 범위 (특정 사용자/전체)

## 워크플로우

1. `references/project-map.md`를 참조하여 이슈 키워드로 관련 프로젝트 특정
2. `references/error-patterns.md`를 참조하여 에러 유형별 탐색 위치 파악
3. `references/architecture.md`를 참조하여 서비스 간 호출 흐름 이해
4. `references/troubleshoot-guide.md`를 참조하여 이슈 유형별 탐색 전략 및 MCP 활용 기준 확인
5. 관련 로직 분석 후 필요 시 lane4-mysql, redis, elasticsearch, firebase MCP 등으로 상세 분석
6. 원인 파악시 이슈 원인 파악 구조에 맞춰 응답 생성
7. 응답을 반복하여 확인 후 정확할 경우 사용자에게 응답
