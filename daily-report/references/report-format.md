# 업무보고 출력 포맷

아래 형식을 정확히 따라 업무보고를 작성한다.

---

## 포맷

```
**[프로덕트팀 지니 {MM/DD} 업무보고]**

{카테고리1}

- {작업내용1}
- {작업내용2}
- {작업내용3}

{카테고리2}

- {작업내용1}
- {작업내용2}

기타

- {기타작업1}
- {기타작업2}
```

## 규칙

### 제목
- 형식: `**[프로덕트팀 지니 MM/DD 업무보고]**`
- MM/DD는 대상 날짜
- 반드시 `**` 볼드 처리한다

### 카테고리
- 프로젝트명 또는 기능 키워드를 대문자/한글로 표기
- 예: `FARE_HISTORY`, `호출자`, `안심번호`, `어드민`, `인스파이어`
- 작업이 1~2개만 있는 단독 주제는 "기타"로 통합 가능
- 카테고리 간 빈 줄로 구분

### 작업 항목
- 한 줄에 하나의 **태스크(작업)**. 커밋 하나하나가 아님
- 각 항목 앞에 `- ` 글머리 기호를 붙인다
- 한국어로 간결하게 작성. 업무 관점으로 쓴다 (코드 변경 설명 X)
- 같은 작업의 여러 커밋은 하나의 항목으로 통합
  - Good: `앱 FARE_HISTORY 이관 및 테스트`
  - Bad: `FARE_HISTORY 엔티티를 실제 DB 테이블 구조와 일치하도록 수정`
- 배포한 항목은 "배포" 또는 "운영 배포 및 모니터링" 명시
- 커밋에 없는 작업도 포함 (원인 파악, 모니터링, 문서화 등)

### 기타
- 특정 프로젝트에 속하지 않는 작업
- 레거시 파악, 문서 정리, 코드리뷰 등

### 퇴근 시간 (선택)
- 사용자가 원할 경우 마지막에 추가
- 형식: `{요일}에는 {MM/DD} ({요일축약}) {HH:MM} 업무 종료합니다. ({출근시간} ~ {퇴근시간})`
- 예: `수요일에는 02/25 (수) 17:28 업무 종료합니다. (08:28 ~ 17:28)`

---

## 실제 예시

```
**[프로덕트팀 지니 02/27 업무보고]**

FARE_HISTORY

- 앱 FARE_HISTORY 이관 및 테스트

인스파이어

- 신규 권한 추가 작업 운영 배포 및 모니터링

기타

- lane4-achakey 환율 로직 스케줄러 이관
- redis, es, firebase mcp 테스트 및 문서화
- 브라이튼여의도 결제 실패 원인 파악 및 웹훅 방식 작업
- 요금 업로드 실패 원인 파악 및 처리
```

```
****[프로덕트팀 지니 02/26 업무보고]****

FARE_HISTORY

- 법인어드민 요금 예외처리 리스트업
- 법인 별 요금 싱크 확인
- 요금 조회 에러로 인한 0원 처리 시 슬랙 알림 추가

기타

- 인스파이어 당일 조건 KST 변환 누락 수정 배포
- 인스파이어 상태변경 신규 권한 추가
```

```
**[프로덕트팀 지니 02/25 업무보고]**

- EK 메일 파싱 테스트
- 어드민 배차 상세 조회 조건 수정 배포
- FARE_HISTORY 업데이트 쿼리 전달
- 안심번호 회수 실패 처리 엔드포인트 실테스트
- FARE_HISTORY 저장 관련 작업
```

```
**[프로덕트팀 지니 02/23 업무보고]**

호출자

- 개발계 테스트
- 알림톡/메일 탑승자/호출자 발송 문서화 및 논의사항 정리
- 누락 알림 수정 배포

기타

- charter 관련 레거시 파악
```
