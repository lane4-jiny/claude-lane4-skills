# Daily Report

당일 git 커밋을 분석하여 업무보고를 작성한다.

## 핵심 제약사항

- 코드를 수정하지 않는다. 리포트 생성만 수행한다.
- git log에 기록된 사실만 기반으로 작성한다. 추측하지 않는다.
- 커밋 메시지가 불명확한 경우 `git diff`를 참고하여 보완한다.
- 출력 형식은 `references/report-format.md`를 반드시 따른다.

| 보고서 이름 | 지니 |
| 팀명 | 프로덕트팀 |

## 워크플로우

### 1단계: 대상 날짜 확인

- 사용자에게 날짜 확인 (기본: 오늘)
- 날짜 형식: MM/DD
- 요일 계산 (출퇴근 시간 기재 시 필요)

### 2단계: git 커밋 조회

```bash
# /Users/ojieun/IdeaProjects/ 내 모든 lane4-* 레포에서 해당 날짜 커밋 조회
for dir in /Users/ojieun/IdeaProjects/lane4-*/; do
  repo_name=$(basename "$dir")
  git -C "$dir" log --author="lane4-jiny" --since="{날짜} 00:00" --until="{날짜+1} 00:00" --oneline 2>/dev/null
done
```

- 각 레포별 커밋을 수집한다.
- 커밋 메시지만으로 작업 내용이 불명확한 경우 `git diff {해시}~1 {해시} --stat`으로 변경 파일을 확인한다.

### 3단계: 작업 항목 분류

커밋을 **태스크(작업) 단위**로 통합하여 분류한다. 커밋 하나하나가 아니라 "무슨 일을 했는가"를 기준으로 묶는다.

분류 기준:
1. **같은 작업의 커밋은 반드시 하나의 항목으로 통합**한다.
   - 예: "feat: FARE_HISTORY 저장 로직 추가" + "fix: FARE_HISTORY 엔티티 수정" → `앱 FARE_HISTORY 이관 및 테스트`
   - 예: "feat: Toss 웹훅 구현" + "refactor: 웹훅 컨트롤러 이관" → `결제 웹훅 방식 작업`
2. **카테고리는 핵심 프로젝트/기능 키워드만 사용** (2개 이상 항목이 있을 때만)
   - 예: FARE_HISTORY, 호출자, 인스파이어, 안심번호
3. **카테고리로 분리할 만큼 항목이 없는 작업은 모두 "기타"에 넣는다.**
4. **커밋에 없는 작업도 사용자에게 확인 후 추가**할 수 있다.
   - 예: 원인 파악, 모니터링, 문서화, 테스트 등 커밋으로 남지 않는 작업

### 4단계: 업무보고 작성

- `references/report-format.md` 형식에 맞춰 출력한다.
- **항목은 태스크 단위로 한 줄 요약.** 코드 변경 사항이 아닌 업무 관점으로 작성한다.
  - Good: `앱 FARE_HISTORY 이관 및 테스트`
  - Good: `신규 권한 추가 작업 운영 배포 및 모니터링`
  - Good: `결제 실패 원인 파악 및 웹훅 방식 작업`
  - Bad: `FARE_HISTORY 엔티티를 실제 DB 테이블 구조와 일치하도록 수정` (너무 상세)
  - Bad: `Toss Payments 결제 완료 웹훅 엔드포인트 구현` (코드 수준 설명)
- "배포"가 포함된 작업은 항목에 "배포" 또는 "운영 배포"를 명시한다.
- "모니터링", "테스트", "원인 파악" 등 후속 작업도 포함한다.

### 5단계: 사용자 검토

- 작성된 업무보고를 사용자에게 보여주고 수정사항이 있는지 확인한다.
- 사용자가 추가/수정 요청 시 반영한다.
- 출퇴근 시간을 기재할지 물어본다.
