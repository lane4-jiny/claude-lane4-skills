# 장애 대응 플레이북

## 심각도 분류

| 심각도 | 기준 | 예시 |
|--------|------|------|
| **Critical** | 서비스 전체 불가, 데이터 유실 | API 서버 다운, DB 장애 |
| **Major** | 핵심 기능 장애, 다수 사용자 영향 | 배차 실패, 결제 오류 |
| **Minor** | 일부 기능 장애, 소수 사용자 영향 | 특정 조건 에러, UI 깨짐 |

## 대응 절차

### Phase 1: 탐지 및 초기 파악 (0~5분)

1. **장애 증상 확인**
   - 어떤 기능이 동작하지 않는가?
   - 언제부터 발생했는가?
   - 에러 메시지가 있는가?

2. **즉시 확인 항목**
   - 최근 배포 이력 확인 (`git log --oneline -10`)
   - 서버 상태 확인 (헬스체크 API)
   - 에러 로그 확인 (DB HTTP_LOG 5xx 필터, ES 에러 로그)

### Phase 2: 로그 분석 및 원인 추적 (5~15분)

1. **에러 로그 검색**
   - DB: `HTTP_LOG`에서 시간대별 5xx 에러 조회
   - ES: 에러/예외 로그 키워드 검색
   - 에러율 추이 확인 (급증 시점 = 장애 시작점)

2. **원인 추적**
   - 에러 스택트레이스 → 코드 위치 추적
   - 최근 배포 변경사항과 비교
   - 외부 의존성 장애 여부 확인 (PG사, 외부 API 등)

3. **영향 범위 파악**
   - 영향받은 API 엔드포인트 목록
   - 영향받은 사용자 수 (유니크 사용자 기준)
   - 비즈니스 임팩트 (배차, 결제 등)

### Phase 3: 대응 (15분~)

1. **즉시 조치 (임시)**
   - 장애 원인이 최근 배포인 경우 → 롤백 검토
   - 특정 기능 장애인 경우 → 해당 기능 비활성화 검토
   - 외부 의존성 장애인 경우 → 대체 로직/재시도 검토

2. **근본 해결**
   - 버그 수정 코드 작성
   - 코드 리뷰 (간소화)
   - 핫픽스 배포

### Phase 4: 사후 처리

1. **장애 보고서 작성**
   - 타임라인 정리
   - 원인 분석 결과
   - 대응 내역
   - 재발 방지 대책

2. **재발 방지**
   - 모니터링 알림 추가/강화
   - 테스트 케이스 보완
   - 코드 방어 로직 추가

## 롤백 절차

### 코드 롤백
```bash
# 1. 롤백 대상 커밋 확인
git log --oneline -20

# 2. 이전 안정 버전 확인
git diff {stable_commit}..HEAD --stat

# 3. 롤백 (사용자가 직접 실행)
# revert 방식 (권장)
git revert {commit_hash}

# 또는 reset 방식 (주의: 이력 삭제)
git reset --hard {stable_commit}
```

### 롤백 전 체크리스트
- [ ] DB 마이그레이션이 포함된 배포인가? (있으면 DB 롤백도 필요)
- [ ] 다른 서비스에 영향을 주는 API 변경이 있는가?
- [ ] 롤백 후에도 데이터 정합성이 유지되는가?

## 자주 쓰는 검색 패턴

### DB (HTTP_LOG)
```sql
-- 시간대별 5xx 에러 조회
SELECT DATE_FORMAT(created_at, '%Y-%m-%d %H:%i') as time_slot,
       COUNT(*) as error_count,
       url
FROM HTTP_LOG
WHERE status_code >= 500
  AND created_at >= NOW() - INTERVAL 1 HOUR
GROUP BY time_slot, url
ORDER BY time_slot DESC;
```

### ES
```
-- 에러 로그 키워드 검색
level: "error" AND @timestamp >= "now-1h"

-- 특정 API 에러
path: "/api/v1/*" AND status: >= 500
```
